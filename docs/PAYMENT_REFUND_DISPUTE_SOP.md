# MELE 付款、退款與申訴 SOP

最後更新：2026-04-30

本文件定義公開收費前必須演練的金流與爭議處理流程。正式條款仍需律師與金流服務商確認。

## 1. 付款流程

1. 使用者選擇老師與服務項目。
2. 系統建立 booking，狀態為 `pending_payment`。
3. 使用者前往 ECPay checkout。
4. ECPay webhook 回傳付款結果。
5. webhook 驗證 CheckMacValue。
6. webhook 呼叫 `confirm_payment`，booking 轉為 `paid`。
7. 使用者回到付款結果頁確認狀態。

必須驗證：

- 同一 webhook 重送不會重複入帳。
- 付款結果頁若 webhook 尚未到達，應顯示「付款狀態確認中」。
- `CustomField1` 或等價欄位必須能追蹤 booking id。

## 2. 撥款與 hold 原則

建議正式營運採用：

- 使用者付款後，平台先 hold 款。
- 諮詢完成後 7 天或申訴期結束後撥款。
- 若有爭議、老師未出席、刷退或異常交易，暫停撥款。
- 老師後台需能看到「待撥款、可撥款、已撥款、爭議中」。

## 3. 取消與退款建議

| 情境 | 建議處理 |
|---|---|
| 使用者於 24 小時前取消 | 全額退款或保留平台手續費，依正式條款 |
| 使用者於 24 小時內取消 | 部分退款，例如 50% |
| 使用者未出席 | 不退款或依老師政策 |
| 老師未出席 | 全額退款，並記錄老師違規 |
| 老師遲到或服務不足 | 依證據部分退款、補時或補償 |
| 金流失敗但 booking 存在 | 保留 `pending_payment`，允許重新付款 |

付款前必須清楚揭露取消與退款規則，並保留使用者同意紀錄。

## 4. 刷退處理

刷退風險：

- 刷退率過高可能導致金流罰款或停用。
- 若款項已撥給老師，平台可能承擔損失。

SOP：

1. 收到刷退通知。
2. 將 booking 標為 `dispute_status = chargeback` 或等價狀態。
3. 暫停該筆撥款與相關 settlement。
4. 收集證據：付款紀錄、預約紀錄、諮詢完成紀錄、雙方訊息。
5. 在金流期限內提交回覆。
6. 完成後更新 booking、settlement 與客服紀錄。

## 5. 申訴流程

使用者可申訴：

- 老師未出席。
- 服務內容與頁面描述不符。
- 老師要求私下付款。
- 老師要求不必要敏感資料。
- 付款、退款或帳務異常。

老師可申訴：

- 使用者未出席。
- 使用者騷擾或辱罵。
- 使用者惡意刷退。
- 使用者要求違規服務。

客服處理 SLA：

- 24 小時內確認收到。
- 3 個工作天內提出初步處理。
- 複雜金流爭議 14 個工作天內更新進度。

## 6. 老師處分規則

| 行為 | 建議處分 |
|---|---|
| 第一次輕微遲到 | 警告並記錄 |
| 多次遲到 | 降低曝光或暫停接案 |
| 未出席 | 退款、警告、暫停接案 |
| 私下導流付款 | 停權或解約 |
| 要求敏感資料 | 停權並人工調查 |
| 詐欺或重大違規 | 立即停權並保留法律追訴 |

## 7. 上線前金流驗收

必須留下證據：

- ECPay sandbox 成功交易編號。
- checkout function log。
- webhook function log。
- booking 狀態從 `pending_payment` 轉 `paid`。
- 付款結果頁截圖。
- 至少一筆取消/退款演練紀錄。
- 重複 webhook 測試紀錄。
